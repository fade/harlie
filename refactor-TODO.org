*** 2. memo.lisp -- Memo store

**** [WARNING] Unbounded growth

=*pending-memos*= is a plain hash table with no size limit. A malicious user
(or just a busy channel) can issue unlimited =!tell= commands. There is no
per-sender limit, no per-recipient limit, and no global cap. Over time this
will consume unbounded memory.

Recommendation: Add a per-recipient cap (e.g. 20 memos) and/or a global cap
(e.g. 1000 total). Reject new memos when the cap is hit.

**** [WARNING] No thread safety

=*pending-memos*= is a bare hash table accessed from multiple threads (the
IRC read thread calls =deliver-memos-for= on join/speak events, while =!tell=
stores memos). SBCL hash tables are not thread-safe for concurrent writes.
This can corrupt the table.

Recommendation: Wrap accesses in a lock, or use a thread-safe structure.

**** [INFO] Purely in-memory -- no persistence

Memos are lost on restart. This is acceptable for a first pass but worth
documenting for users. The code comments say "optional DB persistence" but
none is implemented.

**** [INFO] =deliver-memos-for= removes ALL memos on partial delivery

If delivery of one memo fails (error in =privmsg=), the remaining memos are
still removed from the hash because =remhash= runs unconditionally when
=count > 0=. This means successfully-counted memos plus undelivered ones all
get deleted.

Recommendation: Only remove memos that were actually delivered, or re-store
the ones that failed.

*** 3. !eval sandbox -- SECURITY AUDIT

**** [CRITICAL] Sandbox is trivially escapable

The sandbox checks a banned-symbol list against the parsed form tree, but:

1. =symbol-function=, =fdefinition=, =symbol-value=, =find-symbol=, =intern=,
   =funcall=, =apply=, =progn=, =let=, =flet=, =labels=, =macrolet=,
   =change-class=, =slot-value= are all ALLOWED. An attacker can do:
   #+begin_example
   !eval (funcall (symbol-function 'delete-file) "important.dat")
   !eval (uiop:run-program "rm -rf /" :output :string)
   !eval (funcall (fdefinition 'sb-ext:run-program) "sh" '("-c" "id") :search t :output t)
   #+end_example
   The banned list catches direct use of =sb-ext:run-program= but NOT indirect
   invocation via =funcall=, =apply=, =symbol-function=, or =fdefinition=.

2. =find-symbol= / =intern= can construct any symbol at runtime, completely
   bypassing the name check:
   #+begin_example
   !eval (funcall (intern "RUN-PROGRAM" "SB-EXT") "id" (list) :search t :output t)
   #+end_example

3. =#.= reader macro is correctly disabled (=*read-eval*= is NIL). Good.

4. The timeout wraps both =read-from-string= AND =eval=, which is correct.

5. =make-instance= is banned, but =change-class=, =allocate-instance=,
   =shared-initialize= are not. CLOS can still be abused.

6. =setf= is banned but =set=, =(setf symbol-value)= via =funcall= are not.

Recommendation: This sandbox MUST NOT go to production. A symbol-blacklist
approach is fundamentally flawed in Common Lisp. The only safe approaches are:
(a) a whitelist-only evaluator that walks the form and only permits known-safe
    functions (arithmetic, string ops, list ops), rejecting everything else; or
(b) evaluate in a restricted package with only safe symbols imported; or
(c) remove !eval entirely.

**** [WARNING] Timeout is 5s -- eval runs in the calling thread

The =eval= happens inside =trivial-timeout:with-timeout= in the plugin
dispatch thread. If evaluation triggers an infinite loop that does NOT hit
a safe-point (e.g. a tight CFFI call or a foreign function), the timeout
signal may not be delivered, blocking the plugin dispatcher.

*** 5. !remind plugin

**** [WARNING] Unbounded thread creation

Each =!remind= invocation spawns a new =bt:make-thread= that sleeps for the
requested duration. There is no limit on how many reminders a user can set.
A malicious user can issue thousands of =!remind 86400d msg= commands, each
creating a sleeping thread consuming stack space (typically 1-8 MB per thread).

Recommendation: Cap reminders per-user (e.g. 5) and per-channel, and enforce
a maximum duration (e.g. 7d). Consider using a single scheduler thread with
a priority queue instead of one thread per reminder.

**** [WARNING] Reminder threads hold reference to connection object

If the bot disconnects and reconnects (getting a new connection object), the
sleeping reminder thread still holds the old =conn= reference. The =privmsg=
call will fail or go nowhere.
